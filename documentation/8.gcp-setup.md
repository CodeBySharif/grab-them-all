# 8. GCP Setup

This guide walks through setting up Google Cloud Platform for this project from scratch.

For CI/CD automation, see [9.cicd-pipeline.md](./9.cicd-pipeline.md).  
For custom domains, see [10.custom-domain-and-dns.md](./10.custom-domain-and-dns.md).

---

## Overview

You will create:

| GCP resource | Name | Purpose |
|---|---|---|
| Project | Your project ID | Billing + resource container |
| Artifact Registry | `ferry-flight` | Store Docker images |
| Cloud Run service | `ferry-flight-api` | Backend API |
| Cloud Run service | `ferry-flight-web` | Frontend PWA + nginx |
| Service account | `github-deploy` | GitHub Actions deploy identity |

**Region used:** `asia-southeast1` (Singapore — close to Langkawi/Malaysia)

---

## Prerequisites

- Google account with billing enabled
- [gcloud CLI](https://cloud.google.com/sdk/docs/install) installed (optional for manual steps)
- GitHub repository with this code

---

## Step 1: Create a GCP project

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a new project (e.g. `ferry-flight-prod`)
3. Note the **Project ID** (used everywhere)

Enable billing on the project.

---

## Step 2: Enable required APIs

```bash
export GCP_PROJECT_ID=your-project-id

gcloud config set project $GCP_PROJECT_ID

gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  cloudbuild.googleapis.com \
  iam.googleapis.com
```

| API | Why |
|---|---|
| Cloud Run | Host containers |
| Artifact Registry | Store Docker images |
| Cloud Build | (Optional) Cloud-native builds |
| IAM | Service accounts and permissions |

---

## Step 3: Create Artifact Registry

```bash
export GCP_REGION=asia-southeast1

gcloud artifacts repositories create ferry-flight \
  --repository-format=docker \
  --location=$GCP_REGION \
  --project=$GCP_PROJECT_ID \
  --description="Ferry & Flight containers"
```

Images will be tagged like:

```
asia-southeast1-docker.pkg.dev/PROJECT_ID/ferry-flight/api:COMMIT_SHA
asia-southeast1-docker.pkg.dev/PROJECT_ID/ferry-flight/web:COMMIT_SHA
```

---

## Step 4: Create deployment service account

GitHub Actions needs a service account to deploy:

```bash
gcloud iam service-accounts create github-deploy \
  --display-name="GitHub Actions deploy"

gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="serviceAccount:github-deploy@${GCP_PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/run.admin"

gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="serviceAccount:github-deploy@${GCP_PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/artifactregistry.writer"

gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="serviceAccount:github-deploy@${GCP_PROJECT_ID}.iam.gserviceaccount.com" \
  --role="roles/iam.serviceAccountUser"

gcloud iam service-accounts keys create gcp-sa-key.json \
  --iam-account=github-deploy@${GCP_PROJECT_ID}.iam.gserviceaccount.com
```

> **Best practice:** Prefer [Workload Identity Federation](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines) over long-lived JSON keys for production. JSON keys are simpler for MVP setup.

**Roles explained:**

| Role | Permission |
|---|---|
| `run.admin` | Deploy and update Cloud Run services |
| `artifactregistry.writer` | Push Docker images |
| `iam.serviceAccountUser` | Act as runtime service account during deploy |

## Step 5: Configure GitHub secrets

In GitHub → **Settings → Secrets and variables → Actions**:

| Secret | Value |
|---|---|
| `GCP_PROJECT_ID` | Your GCP project ID |
| `GCP_SA_KEY` | Full contents of `gcp-sa-key.json` |
| `OPENROUTESERVICE_API_KEY` | _(optional)_ ORS API key |

Delete `gcp-sa-key.json` from your local machine after adding to GitHub.

---

## Step 6: First deploy

Push to `main`:

```bash
git push origin main
```

GitHub Actions (`.github/workflows/deploy.yml`) will:

1. Build .NET API and React frontend
2. Build Docker images
3. Push to Artifact Registry
4. Deploy both Cloud Run services

---

## Step 7: Verify Cloud Run services

After deploy, note the default URLs:

```bash
gcloud run services describe ferry-flight-api \
  --region asia-southeast1 \
  --format 'value(status.url)'

gcloud run services describe ferry-flight-web \
  --region asia-southeast1 \
  --format 'value(status.url)'
```

Test:

```bash
curl https://YOUR-API-URL.run.app/api/schedules/ferry
curl https://YOUR-WEB-URL.run.app/
```

## Cloud Run service settings

Set by CI/CD in `deploy.yml`:

| Setting | API | Web |
|---|---|---|
| Port | 8080 | 8080 |
| Memory | 512Mi | 256Mi |
| CPU | 1 | 1 |
| Min instances | 0 | 0 |
| Max instances | 3 | 3 |
| Auth | `--allow-unauthenticated` | `--allow-unauthenticated` |

**Cold starts:** First request after idle may take 2–5 seconds. Set `--min-instances 1` on API if needed for production SLA.

## Manual deploy (without GitHub Actions)

See root `DEPLOYMENT.md` or run:

```bash
export IMAGE_PREFIX=asia-southeast1-docker.pkg.dev/$GCP_PROJECT_ID/ferry-flight

# Build and push API
docker build -t $IMAGE_PREFIX/api:local -f backend/FerryFlight.Api/Dockerfile backend/FerryFlight.Api
docker push $IMAGE_PREFIX/api:local
gcloud run deploy ferry-flight-api --image $IMAGE_PREFIX/api:local --region asia-southeast1 --allow-unauthenticated --port 8080

# Build and push Web
docker build -t $IMAGE_PREFIX/web:local -f frontend/Dockerfile frontend
docker push $IMAGE_PREFIX/web:local
gcloud run deploy ferry-flight-web --image $IMAGE_PREFIX/web:local --region asia-southeast1 --allow-unauthenticated --port 8080 --set-env-vars "BACKEND_URL=https://grab-them-all-api.shariflab.my"
```
