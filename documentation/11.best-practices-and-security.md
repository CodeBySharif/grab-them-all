# 11. Best Practices and Security

A checklist of what this project does well, what to improve, and general lessons for your next project.

---

## Frontend best practices

### What this project does well

| Practice | Where |
|---|---|
| **TypeScript** | Catches type errors at build time |
| **Component split** | One component per UI concern (`TripItem`, `TopBar`, etc.) |
| **Pure logic in utils** | `ferryTiming.ts` has no React — easy to read and test |
| **Context for shared state** | `EstimationContext` avoids prop drilling |
| **Relative API URLs** | `/api/...` works in dev, Docker, and production |
| **Responsive design** | Tailwind breakpoints, mobile-first |
| **Safe area handling** | iPhone notch/home indicator |
| **Separate permission steps** | iOS location vs alerts on different taps |

### Gaps to improve

| Gap | Recommendation |
|---|---|
| No unit tests | Add tests for `ferryTiming.ts` (highest value) |
| No error boundary | Wrap app in React error boundary for crash recovery |
| No loading skeleton per trip | Only grid-level skeleton today |
| Hard-coded route labels | Could come from API if routes change |

> **Takeaway:** Put business rules in pure functions (`utils/`), keep components thin.

## Backend best practices

### What this project does well

| Practice | Where |
|---|---|
| **Dependency injection** | Services registered in `Program.cs`, injected via constructors |
| **Interface segregation** | `IFerryScheduleScraper`, `IFerryScheduleCache`, etc. |
| **Thin controllers** | Controllers delegate to services immediately |
| **Caching** | Reduces load on external ferry website |
| **Stale fallback** | Serves last good data if scrape fails |
| **Config via appsettings + env** | 12-factor app pattern |
| **Input validation** | Lat/lng bounds checked in controller |
| **Structured logging** | `ILogger` for warnings and errors |
| **HttpClientFactory** | Proper HTTP client lifecycle (no socket exhaustion) |

### Gaps to improve

| Gap | Recommendation |
|---|---|
| No automated tests | Add integration tests for scraper + API |
| Scraping fragility | Add health check; alert if parse returns empty |
| No rate limiting | Add middleware if API is public |
| No authentication | OK for public read-only MVP; add if needed later |
| In-memory cache only | Resets on container restart; Redis for multi-instance |

> **Takeaway:** Cache external calls, fail gracefully with stale data, never crash silently.

## Deployment best practices

### What this project does well

| Practice | Where |
|---|---|
| **Multi-stage Dockerfiles** | Small production images |
| **CI builds before deploy** | PR + push validation |
| **Immutable image tags** | Git SHA as Docker tag |
| **Secrets in GitHub** | Not in repo or Dockerfile |
| **Two-service split** | Static web vs compute API |
| **Env-based config** | `BACKEND_URL`, CORS, API keys via env |
| **Infrastructure as code** | `deploy.yml`, Dockerfiles in repo |
| **Documentation** | DEPLOYMENT.md + this folder |

### Gaps to improve

| Gap | Recommendation |
|---|---|
| JSON service account key | Migrate to Workload Identity Federation |
| No staging environment | Add `staging` branch → separate Cloud Run services |
| No rollback automation | Document `gcloud run deploy --image PREVIOUS_TAG` |
| No uptime monitoring | Add UptimeRobot or GCP monitoring alerts |
| Single region | Multi-region only needed at scale |

> **Takeaway:** Automate deploys, never bake secrets into images, tag every release.

## Security best practices

### What this project does well

| Practice | Details |
|---|---|
| **HTTPS everywhere** | Cloudflare + Cloud Run TLS |
| **Secrets not in git** | `.gitignore` for `.env`, keys; GitHub Secrets for CI |
| **CORS scoped** | Only web origin allowed on API |
| **Same-origin API proxy** | Browser calls web domain; reduces CORS exposure |
| **No auth on public read API** | Appropriate for public schedule data |
| **Input validation** | Coordinate bounds on estimation endpoint |
| **Non-root containers** | Cloud Run default security |
| **allow-unauthenticated** | Intentional for public app; document if adding auth later |

### Security checklist for production

| Item | Status | Action if needed |
|---|---|---|
| HTTPS | ✅ | — |
| Secrets in env/secrets manager | ✅ | Rotate SA key periodically |
| CORS restricted | ✅ | Update on domain change |
| API rate limiting | ❌ | Add if abused |
| User authentication | ❌ | Not needed for MVP |
| SQL injection | N/A | No database |
| XSS | ✅ React escapes by default | Avoid `dangerouslySetInnerHTML` |
| Dependency scanning | ❌ | Enable Dependabot on GitHub |
| Scraped content trust | ⚠️ | Ferry site HTML is parsed; sanitize if displaying raw HTML (currently text only) |

### Secrets — never commit

```
.env
gcp-sa-key.json
OpenRouteService API keys
Any password or token
```

Use:

- `.env.example` — template without real values
- GitHub Secrets — for CI/CD
- Cloud Run env vars — for runtime config
- `dotnet user-secrets` — for local backend dev

> **Takeaway:** Treat the repo as public. All secrets live outside git.

## Architecture lessons (for your next project)

1. **Start with monorepo** for small full-stack apps — simpler than two repos.
2. **Proxy API through nginx** — one origin for the browser.
3. **Scrape + cache + stale fallback** — pragmatic when no official API exists.
4. **Pure functions for business rules** — `ferryTiming.ts` pattern scales well.
5. **Deploy early** — Cloud Run makes production cheap and real feedback valuable.
6. **Document as you go** — this folder exists because the project grew fast via AI.

## Suggested learning exercises

To truly understand this codebase, try:

1. **Change cache TTL** — `FerrySchedule:CacheMinutes` in appsettings, observe behavior
2. **Add a console.log** in `FerryScheduleScraper` — see raw scrape in API logs
3. **Modify Yes/No rule** — edit `evaluateArrival()` in `ferryTiming.ts`
4. **Break and fix deploy** — intentionally wrong `BACKEND_URL`, then fix via workflow
5. **Read one request end-to-end** — browser DevTools Network tab while loading app
6. **Add a unit test** for `parseDepartureDateTime()` — first step toward test coverage
