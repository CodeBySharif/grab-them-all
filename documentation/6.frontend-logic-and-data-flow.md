# 6. Frontend Logic and Data Flow

## User journey (end to end)

```
1. Open app
2. See schedule grid (loading → 3 days of ferries)
3. Tap "Enable Location"
4. Browser asks for GPS permission
5. App fetches travel time to Langkawi terminal
6. Each trip shows Yes/No + leave-by / arrival window
7. Tap "Enable Alerts" (separate step on iOS)
8. Urgent trips show bottom toasts + optional push notifications
```

---

## Data flow diagram

```
App.tsx
  │
  ├─ useEffect → fetchFerrySchedules() → GET /api/schedules/ferry
  │                                      → setSchedules(data)
  │
  └─ EstimationProvider
        │
        ├─ requestLocation()
        │     → navigator.geolocation.getCurrentPosition()
        │     → fetchTravelTimes(lat, lng) → GET /api/estimation/travel-times
        │     → setTravelSeconds, state = 'ready'
        │
        └─ evaluateTrip(route, date, time)
              → evaluateArrival() in ferryTiming.ts
              → 'yes' | 'no' | 'past' | 'unavailable'
```

## Schedule display logic

### Route grid layout (`routes.ts`)

Four fixed slots in this order:

| Position | Route | Direction |
|---|---|---|
| Top-left | Kedah — Langkawi | Inbound |
| Top-right | Perlis — Langkawi | Inbound |
| Bottom-left | Langkawi — Kedah | Outbound |
| Bottom-right | Langkawi — Perlis | Outbound |

Backend may return routes in any order; `orderRoutesForGrid()` maps them to this layout.

### Trip slots (`scheduleLayout.ts`)

Each route shows **exactly 5 slots**. If the ferry site has fewer trips, empty slots show `TripSlotPlaceholder`.

## Driver timing logic (`ferryTiming.ts`)

This file is the **heart of the app**. Read it carefully.

### Route types

| Type | Routes | Driver job |
|---|---|---|
| **Inbound** | Kedah/Perlis → Langkawi | Pick up passenger after ferry arrives |
| **Outbound** | Langkawi → Kedah/Perlis | Drop off passenger before ferry leaves |

### Inbound rules

- **Ferry crossing time:**
  - Kedah route: 90–120 minutes after departure
  - Perlis route: 60–90 minutes after departure
- **Display:** "Ferry arrives 10:30 AM – 12:00 PM"
- **Leave by:** ferry arrival earliest − your drive time
- **Yes/No:** If you leave now, can you reach pickup point by earliest arrival?
- **Closed:** After latest arrival window passes

### Outbound rules

- **Terminal deadline:** 1 hour before departure ("Be around terminal at 5:00 PM" for 6:00 PM ferry)
- **Leave by:** terminal deadline − drive time
- **Yes/No:** Can you reach terminal by **ferry departure time** (not just 1 hr before)?
  - Drivers can still accept jobs until the ferry actually leaves
- **Closed:** After ferry departure time

### Priority levels

| Priority | Meaning |
|---|---|
| `urgent` | ≤ 15 min to event or leave-by time passed |
| `soon` | ≤ 30 min |
| `normal` | Plenty of time |
| `past` | Trip window closed |

## Yes/No indicator (`ArrivalIndicator.tsx`)

Driven by `evaluateArrival()`:

```typescript
// Simplified logic
etaAtTerminal = now + travelSeconds
if (outbound) {
  return etaAtTerminal <= departure ? 'yes' : 'no'
} else {
  return etaAtTerminal <= earliestArrival ? 'yes' : 'no'
}
```

Shown only when:

- Location is enabled (`state === 'ready'`)
- Trip is not past (`isTripEstimationPast`)

## Priority alerts flow

```
PriorityAlerts
  │
  ├─ useEstimation() → travelSeconds, state
  ├─ useNowTick() → current time (updates every minute)
  ├─ collectPriorityAlerts(days, travelSeconds, now)
  │     → filters urgent/soon trips
  ├─ PriorityAlertToast → shows up to 3 toasts at bottom
  └─ usePriorityNotifications → browser Notification API for urgent
```

## Permission flow (iOS vs Android)

| Step | Action | iOS note |
|---|---|---|
| 1 | Tap **Enable Location** | Must be direct user tap |
| 2 | Geolocation prompt | Works in Safari and PWA |
| 3 | Travel time loads | Banner shows drive duration |
| 4 | Tap **Enable Alerts** | **Separate button** — iOS ignores delayed/auto prompts |
| 5 | Notification prompt | Push only works from Home Screen PWA on iOS 16.4+ |

`platform.ts` detects iOS, standalone PWA, and notification support.

> **Best practice:** Never call `Notification.requestPermission()` inside `setTimeout` or geolocation callbacks on iOS. Use a separate button click.

## State management approach

| Data | Where stored | Why |
|---|---|---|
| Schedules | `App.tsx` useState | Only root needs it |
| Travel/location | `EstimationContext` | Many child components need it |
| Dismissed alerts | `PriorityAlerts` useState | Local UI state |
| Current time | `useNowTick` hook | Shared tick without prop drilling |

No Redux — the app is small enough for Context + local state.
