# 1. Overview and Tech Stack

## Product overview

**Ferry & Flight PWA** helps drivers and passengers around Langkawi check ferry schedules and decide if they can reach the terminal in time.

### Primary users

- **Taxi/grab drivers** dropping off or picking up passengers for ferries.
- **Passengers** checking whether they can still catch a specific sailing.

### Core features (ferry module)

| Feature | Description |
|---|---|
| Schedule grid | 3 days × 4 routes, 5 fixed trip slots per route |
| Travel estimation | Drive time from your GPS to Langkawi terminal |
| Yes/No indicator | Can you arrive before the relevant deadline? |
| Leave-by time | When to start driving |
| Priority alerts | Urgent/soon warnings via toasts and push notifications |
| Offline PWA | Last fetched schedules cached in the browser |
| Installable | Add to Home Screen (iOS/Android) |

### Product stage

This is **MVP+** (version 1 for a niche market):

- More than a demo — it is deployed, has custom domains, and real scraping.
- Less than a full platform — no user accounts, no database, no flight module yet, no automated tests.

| Setting | Where | Default |
|---|---|---|

| Web → API URL | `BACKEND_URL` env (Docker/Cloud Run) | — |

## Why these technologies?

| Choice | Reason |
|---|---|
| React + Vite | Industry standard, fast development, great PWA support |
| ASP.NET Core | Strong HTTP/API tooling, easy Docker deployment, good for scraping + caching |
| Monorepo | One repo, one pipeline — frontend and backend ship together |
| Cloud Run | No server management, scales to zero, pay per request |
| Scraping vs official API | Ferry site has no public API; scraping is the practical option |
| nginx proxy | Browser calls same origin (`/api`); avoids CORS complexity |

> **Best practice:** Pick boring, proven tools for MVPs. React + a small API + Cloud Run is a common, maintainable stack.
