# 2. Monorepo and Architecture

## What is a monorepo?

A **monorepo** means one Git repository contains multiple related projects:

```
grab-them-all/
├── backend/     ← API
├── frontend/    ← PWA
└── .github/     ← Shared CI/CD
```

### Why this project uses a monorepo

| Benefit | How it helps here |
|---|---|
| **Single source of truth** | One place for docs, Docker, and deploy config |
| **Atomic deploys** | Push once → both API and web rebuild together |
| **Easier local dev** | `docker-compose.yml` runs full stack with one command |
| **Shared versioning** | Frontend and backend stay in sync |

### When a monorepo is NOT ideal

- Large teams with independent release cycles per service.
- Unrelated apps that share nothing.

For a small two-service app like this, monorepo is the right choice.

## Request flow (typical page load)

1. User opens `https://grab-them-all.shariflab.my`
2. Cloudflare terminates SSL, forwards to Cloud Run **web** service
3. nginx serves `index.html`, JS, CSS from the Docker image
4. React app loads and calls `GET /api/schedules/ferry` (relative URL)
5. nginx proxies that request to `BACKEND_URL` (API custom domain)
6. API checks memory cache → scrapes ferry site if needed → returns JSON
7. React renders the 3-day schedule grid
8. User taps **Enable Location** → browser GPS → `GET /api/estimation/travel-times`
9. Frontend computes Yes/No per trip using `ferryTiming.ts`

## Why two Cloud Run services?

| Service | Role | Why separate |
|---|---|---|
| `ferry-flight-web` | Static files + nginx proxy | Lightweight, cheap, CDN-friendly |
| `ferry-flight-api` | .NET API, scraping, external calls | Heavier, different scaling needs |

Users only need the **web URL**. The API URL exists for direct access, CORS, and nginx proxying.

> **Best practice:** Split static frontend and API when the API does heavy work (scraping, third-party APIs). nginx on the web tier keeps the browser on one origin.

## Local development architecture

### Option A — Separate processes

```bash
# Terminal 1
cd backend/FerryFlight.Api && dotnet run    # :5000

# Terminal 2
cd frontend && npm run dev                  # :5173, proxies /api → :5000
```

Vite dev server proxies `/api` to the backend (`frontend/vite.config.ts`).

### Option B — Docker Compose

```bash
docker compose up --build
```

| Service | URL |
|---|---|
| Web | http://localhost:8081 |
| API | http://localhost:8080 |

`docker-compose.yml` sets `BACKEND_URL=http://api:8080` for the web container.

---

## Key environment variables

| Variable | Service | Purpose |
|---|---|---|
| `BACKEND_URL` | web (nginx) | Where to proxy `/api` requests |
| `PORT` | api | Cloud Run injects this; API listens on 8080 |
| `Cors__Origins__0` | api | Allowed browser origin for direct API calls |
| `OpenRouteService__ApiKey` | api | Optional road routing API key |
| `ASPNETCORE_ENVIRONMENT` | api | `Production` in cloud |

## Files that define architecture

| File | Role |
|---|---|
| `docker-compose.yml` | Local two-service stack |
| `frontend/Dockerfile` | Build React → nginx image |
| `frontend/nginx.conf.template` | Static files + API proxy rules |
| `frontend/docker-entrypoint.sh` | Inject `BACKEND_URL` into nginx config |
| `backend/FerryFlight.Api/Dockerfile` | Build .NET API image |
| `.github/workflows/deploy.yml` | Deploy both services to GCP |
